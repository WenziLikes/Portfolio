# Deployment

## Deployment Model

The site is built as a static React application with prerendered route HTML and can be deployed to any host that supports:

- static file serving
- directory index resolution for route folders
- custom `404.html` handling
- HTTPS on the final domain

The deployment unit is the contents of `build/`.

## Build Command

```bash
npm run build
```

That command performs:

1. TypeScript type-checking
2. Vite client build
3. SSR entry build for prerendering
4. static route prerendering through `scripts/prerender.mjs`
5. generation of `robots.txt`, `sitemap.xml`, `/home` redirect HTML, and `404.html`

## Build Output

The final build includes:

- `build/index.html`
- `build/about/index.html`
- `build/expertise/index.html`
- `build/experience/index.html`
- `build/projects/index.html`
- `build/resume/index.html`
- `build/privacy/index.html`
- `build/copyright/index.html`
- `build/canada/index.html`
- `build/usa/index.html`
- `build/europe/index.html`
- `build/home/index.html`
- `build/404.html`
- hashed CSS and JS bundles
- copied static assets from `public/`
- `build/robots.txt`
- `build/sitemap.xml`
- `build/manifest.json`
- `build/documents/viacheslav-murakhin-resume.pdf`

## Hosting Requirement

Because the app ships prerendered HTML for known routes, direct requests to routes such as:

- `/about`
- `/expertise`
- `/experience`
- `/projects`
- `/resume`
- `/privacy`
- `/copyright`
- `/canada`
- `/usa`
- `/europe`

should resolve to their generated HTML files without relying on a catch-all SPA rewrite.

`/home` should serve the generated redirect page, and unknown routes should return `404.html`.

## Local Release Validation

Run before deployment:

```bash
npm install
npm test
npm run build
npm run test:e2e
```

If the UI changed materially, also refresh docs screenshots:

```bash
npm run docs:screenshots
```

If resume content changed, also regenerate the PDF:

```bash
npm run export:resume
```

Then preview locally:

```bash
npm run preview
```

Preview address:

```text
http://localhost:4173
```

## Resume Export

The public resume PDF is generated by:

```bash
npm run export:resume
```

This workflow:

1. builds the app
2. starts a temporary static server against `build/`
3. opens `/resume` in Playwright Chromium
4. exports the PDF into `public/documents/`

If Chromium is missing, install it with:

```bash
npm run test:e2e:install
```

## Domain and Metadata Checklist

Before publishing to the live domain:

1. confirm the canonical URL matches the production domain
2. confirm `robots.txt` points to the correct sitemap URL
3. confirm Open Graph and Twitter preview image paths resolve publicly
4. confirm the downloadable resume PDF path resolves publicly
5. confirm deep links load directly on the host
6. confirm the regional pages expose the intended `hreflang` alternates
7. confirm HTTPS is enabled
8. if `VITE_GA_MEASUREMENT_ID` is set, confirm the correct GA4 property is used

## Host Notes

### Vercel

- publish the generated `build/` output
- verify the production domain is the canonical one

### Netlify

- set the publish directory to `build`
- confirm direct route handling resolves the generated route folders correctly

### Generic static host

- upload the contents of `build/`
- preserve directory index resolution
- preserve static file serving for `documents/`, icons, and preview assets
- let unknown routes return `404.html`

## Post-Deploy Validation

After the site is live:

1. open `/`
2. open `/resume` directly
3. open `/privacy` directly
4. open `/copyright` directly
5. open `/canada`, `/usa`, and `/europe` directly
6. open `/about`, `/expertise`, `/experience`, and `/projects` directly
7. open `/home` and confirm it redirects to `/`
8. verify the home hero `Resume` button opens `/resume`
9. verify the footer and `/resume` page download the current PDF
10. verify favicon, manifest, and social preview assets
11. if analytics is enabled, verify the first visit appears in GA4 Realtime
12. re-submit the sitemap in search tooling if needed
